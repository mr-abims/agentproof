# @agentproof/contracts

The AgentProof Compact contract — three ZK circuits, two ledger maps, and the TypeScript witness driver that pairs with them.

## Layout

```
contracts/
  src/
    agentproof.compact      Compact source (ZK circuits + ledger state)
    witnesses.ts            TS implementation of the witnesses (getReputation, issuerSecret)
    index.ts                Re-exports bindings + witnesses for downstream packages
    managed/                Generated by `compact compile` — gitignored
      agentproof/
        contract/           TS bindings (Contract, Ledger, types, ...)
        keys/               Proving + verification keys (3 circuits × 2)
        zkir/               Zero-knowledge intermediate representations
        compiler/           Compiler metadata
  package.json
  tsconfig.json
```

## Prerequisites

### Install the Compact toolchain

```sh
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/midnightntwrk/compact/releases/latest/download/compact-installer.sh \
  | sh
source ~/.zshrc      # or ~/.bashrc
compact update
```

Verify:

```sh
compact --version
compact compile --version    # want 0.31.0+ (language version 0.23.0+)
```

Older compilers may not understand the `disclose()` requirement or the `Map.member` method.

### Docker (only for generating proofs against testnet)

The Compact compiler does not need Docker. The Midnight proof server does. See [`PHASE4_SETUP.md`](../PHASE4_SETUP.md) for the proof-server setup.

## Build

From this directory:

```sh
npm install
npm run compile:compact     # .compact → src/managed/agentproof/
```

The first compile takes the longest (~60s) because it generates proving and verification keys for all three circuits. Subsequent compiles only re-run when the circuit changes.

## What the contract exposes

Three circuits, callable from the TypeScript bindings:

| Circuit | Public inputs | What it proves / does |
|---|---|---|
| `registerAgent` | `agentId`, `category` | Records a new agent. The issuer identity stored on chain is `persistentHash("agentproof:issuer:v1", issuerSecret)` — derived from the witness, not passed in. |
| `submitVerification` | `agentId`, `minCompletedTasks`, `minSuccessRate`, `minSafetyScore`, `requireNoActiveSlashes` | Asserts the caller is the agent's registered issuer, then proves (in ZK) the private reputation satisfies every threshold, then writes the public `Verification` record. |
| `revokeVerification` | `agentId` | Asserts the caller is the agent's registered issuer, then flips `verified: false`, `revoked: true`. |

Ledger state (`agents`, `verifications`) is `export ledger`, so the frontend / CLI reads it directly via the indexer — no "get" circuit needed.

## Witnesses (the private side)

Two declarations in the Compact source, both implemented in `src/witnesses.ts`:

```compact
witness getReputation(agentId: Field): ReputationData;
witness issuerSecret(): Bytes<32>;
```

`getReputation` returns the agent's private reputation tuple (`completedTasks`, `successfulTasks`, `safetyScore`, `activeSlashes`). The TS driver looks it up in `PrivateState.reputation` keyed by the agent id.

`issuerSecret` returns the 32-byte secret that identifies this DApp instance as a specific issuer. The contract derives `persistentHash("agentproof:issuer:v1", secret)` inside the proof and binds it to the agent at register time; subsequent verify / revoke calls re-derive it and assert equality.

## Privacy property

The exact reputation values (`completedTasks`, `successfulTasks`, `safetyScore`, `activeSlashes`) live only in `PrivateState` and the proof transcript. They are never:

- written to the ledger,
- returned from an exported circuit,
- visible to anyone other than the agent.

If any threshold check fails — or the auth assert fails — no proof is generated and no transaction is broadcast. The public ledger only ever contains attested, issuer-authorized verifications.

## Authorization property

Only the holder of an agent's `issuerSecret` can submit or revoke a verification for that agent. The assertion lives at the top of both circuits:

```compact
assert(agent.issuer == issuerPublicKey(issuerSecret()),
  "caller is not the registered issuer");
```

A different CLI run with a different `issuerSecret` will fail proof generation at this assert — the proof server returns no proof, no transaction is produced. See [spec §16.3](../AGENTPROOF_V1_SPEC.md#163-threat-model-and-what-a-judge-can-verify) for the full audit table and the how-to-verify steps.

## Witness sanity asserts

`submitVerification` validates the witness output before trusting it in the threshold math:

```compact
assert(rep.successfulTasks <= rep.completedTasks, "successfulTasks exceeds completedTasks");
assert(rep.safetyScore <= 100, "safetyScore out of range");
```

Without these, a dishonest issuer could submit impossible private state (e.g. `successful=100`, `completed=10`) and pass any success-rate threshold. The asserts run inside the proof, so they cannot be skipped.

## What's intentionally NOT on chain

- `agentName` — UI-only metadata, not load-bearing.
- `verifiedAt` — the indexer records the containing tx's block timestamp, which is the canonical source.
- `issuerId` strings — only the hash of the issuer secret. The human-readable name lives in client-side demo data.
- `exists: Boolean` — `Map.member(k)` is sufficient.
